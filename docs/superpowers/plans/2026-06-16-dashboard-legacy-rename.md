# Dashboard Legacy Rename — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. This is a **rename**, so tasks use build/test/grep verification gates rather than TDD red-green.

**Goal:** Move the existing dashboard into a `dashboard-legacy` workspace on its current port :5173, keeping it fully functional (dev, prod, e2e, lint), so the `dashboard/` name and the :5174 slot are free for the new app.

**Architecture:** Pure rename. `git mv dashboard dashboard-legacy`, rename the npm package `@housingchoice/dashboard` → `@housingchoice/dashboard-legacy`, and repoint every functional reference (root workspaces, dev launcher, Dockerfile, lint ignore, static-smoke test path) at the new location. No runtime/behavior change. Port stays 5173 (hardened with `strictPort: true`).

**Tech Stack:** npm workspaces, Vite 7, Docker (multi-stage), Vitest, Playwright (e2e).

**Spec:** [docs/superpowers/specs/2026-06-16-dashboard-legacy-rename-design.md](../specs/2026-06-16-dashboard-legacy-rename-design.md)

**Execution model (per user):** Orchestrated with subagents — a **builder** does the edits, a **reviewer** checks correctness/completeness, and a final **adversarial reviewer** hunts security issues, missed renames, and race conditions. The orchestrator (me) is responsible for triaging adversarial findings and dropping anything pedantic. See "Execution Handoff" at the bottom.

**Isolation:** Do this on a dedicated branch/worktree (via superpowers:using-git-worktrees) — it touches root-level structural files; do not move `main`'s HEAD. Hand off for merge per the usual workflow.

## Execution Handoff

**Plan complete.** Per the user's directive, execute this **orchestrated with subagents**, not inline:

1. **Builder subagent** — executes Tasks 1–6 in order (fresh, focused context), committing per task, running each task's verification gate.
2. **Reviewer subagent** (superpowers:requesting-code-review style) — checks the diff for correctness, completeness, and that every File-Map row landed; confirms the verification gates actually passed (evidence, not assertion).
3. **Adversarial reviewer subagent** — explicitly hunts:
   - **Security:** did the rename expose/disable any guard (origin-secret, CSP, CSRF, dev-only `__dev`/`dev-login` gating)? Any secret/path now world-served?
   - **Missed renames:** any remaining functional `@housingchoice/dashboard` / `dashboard/<path>` reference that breaks dev, prod (Docker), lint, tests, or e2e — including ones this plan's File Map didn't anticipate.
   - **Race conditions / unforeseen coupling:** port collisions (5173 vs future 5174, `strictPort`), the `concurrently` `killOthersOn` lifecycle, npm workspace-resolution / lockfile drift, the static-smoke test's self-skip masking a real break, prod `DASHBOARD_DIST_DIR` path assumptions.

**Orchestrator responsibility (me):** I verify each adversarial finding is a *real* issue before acting on it. Pedantic/nitpicky findings (e.g. cosmetic comment wording in unrelated files) I drop with a one-line rationale. Confirmed issues get fixed (by a follow-up builder pass) and re-verified before the branch is handed off for merge.
